## 第一章 · 概述

Agent Memory Manager 是一个 **AI Agent 三层记忆管理微服务**。它把记忆管理从业务逻辑中解耦，提供自动化的长短期记忆注入接口。

### 三层记忆模型

```
Agent 记忆    → agents + skills          → system prompt（人设 + 技能）
User 记忆     → memory_fragments         → 语义检索注入（top-K）
Session 记忆  → sessions + messages      → 最近 N 轮对话上下文
```

每次调用上下文接口时，三层记忆自动聚合为一个结构化的 `Context`，可直接拼装为 LLM 消息列表。

### 核心功能

| 功能 | 说明 |
|------|------|
| Agent / Skill 管理 | 注册 Agent（人设 + 配置）、创建 Skill（关键词触发 + 提示片段）、绑定与解绑 |
| Session / Message 管理 | 创建会话、追加消息、查询历史 |
| LLM 增强（可选） | 自动摘要生成、用户记忆提取、向量嵌入，未配置时降级 |
| pgvector 语义检索 | 余弦相似度 + 全文检索双模式 |

## 第二章 · 部署

### 2.1 Docker 一键启动

```bash
git clone https://github.com/NullWinters/AgentMemoryManager.git
cd AgentMemoryManager
docker compose up -d
```

启动后包含两个容器：

| 容器 | 端口 | 镜像 |
|------|:---:|------|
| `db` | 5432 (内部) / 5433 (宿主机) | `pgvector/pgvector:pg16` |
| `api` | 8000 (容器内) / **8080** (宿主机，可用 `API_PORT` 改) | 本地构建 (FastAPI) |

### 2.2 验证部署

```bash
curl http://localhost:8080/api/v1/health
```

预期响应：

```json
{"status": "ok", "pgvector": true}
```

### 2.3 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DATABASE_URL` | `postgresql+asyncpg://memory:memorypass@db:5432/memorydb` | PostgreSQL 连接串 |
| `API_PORT` | `8080` | Docker 宿主机映射端口（容器内仍为 8000） |
| `MEMORY_API_KEY` | `""` (空) | API 认证 Key，为空时跳过验证 |

## 第三章 · 快速入门

以下示例全部使用 `curl`，无需安装 SDK。

### 3.1 无 LLM 的规则模式（最简单）

**① 注册 Agent**

```bash
curl -X POST http://localhost:8080/api/v1/agents \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "book_bot",
    "name": "书籍推荐助手",
    "persona": "你是书籍推荐助手，语气亲切、专业...",
    "config": {"max_session_turns": 20}
  }'
```

**② 创建 Skill 并绑定**

```bash
curl -X POST http://localhost:8080/api/v1/skills \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Python进阶书单",
    "trigger_keys": ["Python", "进阶", "书单"],
    "prompt_snippet": "当用户需要进阶Python书籍时，推荐《Effective Python》..."
  }'
```

响应中取得 `skill_id`，然后绑定：

```bash
curl -X POST http://localhost:8080/api/v1/agents/book_bot/skills/{skill_id}
```

**③ 创建 User + 开始 Session**

```bash
# 创建 Session（User 不存在时自动创建）
curl -X POST http://localhost:8080/api/v1/sessions \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "book_bot",
    "user_id": "zhang",
    "session_id": "sess_01"
  }'
```

**④ 对话循环**

```bash
# 追加用户消息
curl -X POST http://localhost:8080/api/v1/sessions/sess_01/messages \
  -H "Content-Type: application/json" \
  -d '{"role": "user", "content": "推荐一本进阶 Python 书"}'

# 查询消息历史
curl http://localhost:8080/api/v1/sessions/sess_01/messages?limit=20
```

### 3.2 启用 LLM（完整模式）

在注册 Agent 时配置 `config.llm` 和 `config.embedding`：

```bash
curl -X POST http://localhost:8080/api/v1/agents \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "bot",
    "name": "智能助手",
    "persona": "你是智能助手...",
    "config": {
      "max_session_turns": 20,
      "llm": {
        "api_key": "sk-xxx",
        "base_url": "https://api.deepseek.com",
        "model": "deepseek-chat"
      },
      "embedding": {
        "api_key": "BBFTQJUKU289A4MIZ672GAINCIFULKXO5ALLWA14",
        "base_url": "https://ai.gitee.com/v1",
        "model": "Qwen3-Embedding-8B"
      }
    }
  }'
```

此后每次添加消息，系统会自动：

1. **生成 embedding** — 写入 `messages.embedding` 列
2. **超阈值时生成摘要** — 消息数超过 40 条时 LLM 生成对话摘要
3. **提取用户记忆** — LLM 提取偏好/事实，写入 `memory_fragments` 和 `users.profile`

## 第四章 · API 参考

> Base URL: `http://localhost:8080/api/v1`
> 所有请求体为 JSON，`Content-Type: application/json`

### 4.1 Agent 管理

#### POST /agents — 注册 Agent

```bash
curl -X POST http://localhost:8080/api/v1/agents \
  -H "Content-Type: application/json" \
  -d '{"agent_id":"bot","name":"助手","persona":"...","config":{"max_session_turns":20}}'
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| `agent_id` | string | ✅ | 唯一 ID |
| `name` | string | ✅ | 显示名称 |
| `persona` | string | | 人设描述，默认 `""` |
| `config` | dict | | 配置（含 `llm` / `embedding` 子块），默认 `{}` |

响应 `200`:

```json
{"agent_id":"bot","name":"助手","persona":"...","config":{},"status":"active","created_at":"..."}
```

| 状态码 | 说明 |
|:---:|------|
| 200 | 成功 |
| 409 | Agent 已存在 |

#### GET /agents/{agent_id} — 获取 Agent

```bash
curl http://localhost:8080/api/v1/agents/bot
```

| 状态码 | 说明 |
|:---:|------|
| 200 | 成功 |
| 404 | 不存在 |

#### PATCH /agents/{agent_id} — 更新 Agent

```bash
curl -X PATCH http://localhost:8080/api/v1/agents/bot \
  -H "Content-Type: application/json" \
  -d '{"status":"inactive","name":"新名称"}'
```

所有字段可选：`name`, `persona`, `config`, `status`。未传字段保持不变。

#### Skill 绑定

| 方法 | 路径 | 说明 | 状态码 |
|------|------|------|:---:|
| POST | `/agents/{agent_id}/skills/{skill_id}` | 绑定 Skill | 201 / 404 / 409 |
| DELETE | `/agents/{agent_id}/skills/{skill_id}` | 解绑 Skill | 204 / 404 |
| GET | `/agents/{agent_id}/skills` | 查看已绑定 Skills | 200 / 404 |

```bash
# 绑定
curl -X POST http://localhost:8080/api/v1/agents/book_bot/skills/{skill_id}

# 查看（含 enabled 状态）
curl http://localhost:8080/api/v1/agents/book_bot/skills

# 解绑
curl -X DELETE http://localhost:8080/api/v1/agents/book_bot/skills/{skill_id}
```

## 第五章 · LLM 配置指南

### 5.1 配置结构

`agents.config` 中有两个独立配置块：

```json
{
  "config": {
    "llm": {
      "api_key": "sk-xxx",
      "base_url": null,
      "model": "gpt-4o-mini"
    },
    "embedding": {
      "api_key": "sk-yyy",
      "base_url": null,
      "model": "text-embedding-3-small"
    }
  }
}
```

- **`llm`** — Chat 模型（用于会话摘要和用户记忆提取）
- **`embedding`** — 文本转向量模型（用于语义检索；**不配置时 fallback 到 `llm`**）

### 5.2 三种配置模式

| 模式 | `llm` | `embedding` | 行为 |
|------|:---:|:---:|------|
| 规则模式 | ❌ | ❌ | 无 LLM，仅存储消息，embedding 为 NULL |
| 共享模式 | ✅ | ❌ (fallback) | Chat 和 Embedding 共用同一 Provider |
| **分离模式** | ✅ | ✅ | Chat 和 Embedding 各自独立 Provider |

### 5.3 支持的 Provider

任何兼容 OpenAI API 格式的服务均可使用（`POST /v1/chat/completions` + `POST /v1/embeddings`）。

| Provider | Chat | Embedding | base_url |
|----------|:---:|:---:|------|
| OpenAI | ✅ | ✅ | `https://api.openai.com` |
| DeepSeek | ✅ | ❌ | `https://api.deepseek.com` |
| Azure OpenAI | ✅ | ✅ | `https://xxx.openai.azure.com` |
| Ollama (本地) | ✅ | ✅ | `http://localhost:11434/v1` |
| GITEE AI (模力方舟) | ✅ | ✅ | `https://ai.gitee.com/v1` |

### 5.4 降级行为

| LLM 状态 | Messages | Embedding | Summary | Memory Extraction |
|----------|:---:|:---:|:---:|:---:|
| `llm.api_key` 为空 | ✅ 正常存储 | — | — | — |
| `llm.api_key` 有效 | ✅ | ✅ (如配置) | ✅ (≥41条) | ✅ (≥41条) |
| Embedding API 失败 | ✅ | NULL (降级) | ✅ | ✅ |
| Chat API 失败 | ✅ | ✅ | — | — |

## 第七章 · 常见问题

### Q: 为什么消息发出后 `messages.embedding` 是 NULL？

A: 检查 Agent 的 `config.llm.api_key` 和 `config.embedding.api_key` 是否已配置且有效。另外，某些 Provider（如 DeepSeek）不支持 embedding API，embedding 会优雅降级为 NULL。

### Q: 为什么 `VECTOR` 列没有固定维度？

A: 不同 Provider 返回不同维度（OpenAI=1536, GITEE AI=1024）。系统使用无维度约束的 `VECTOR` 类型，兼容任意维度的 embedding。

### Q: 如何接入本地 Ollama 模型？

A: Agent 配置中设置 `base_url` 为 Ollama 地址：

```json
{
  "llm": {
    "api_key": "ollama",
    "base_url": "http://localhost:11434/v1",
    "model": "qwen2.5"
  },
  "embedding": {
    "api_key": "ollama",
    "base_url": "http://localhost:11434/v1",
    "model": "nomic-embed-text"
  }
}
```

### Q: 消息数达到多少时会触发摘要和记忆提取？

A: 当前阈值固定为 **40 条消息**。超过阈值后，每次添加消息都会在后台生成摘要（写入 `sessions.summary`）和提取用户记忆（写入 `memory_fragments` 和 `users.profile`）。

### Q: 如何查看 LLM 调用日志？

A: LLM 调用日志输出到应用标准日志（`logger.warning` 级别）。调试前端（开发中）将提供可视化的 LLM Logs Tab。
